name: md-writer
description: "Write any .md file safely. For long or complex documents, splits the content into multiple topic-based files and notifies the user before writing. Prevents token overflow errors from single-call large file writes."
allowed-tools:
  - Read
  - Write
  - Edit
  - Glob
---

## 역할

모든 `.md` 파일 작성 시 호출하는 스킬.
한 번의 `Write` 호출로 수만 토큰짜리 파일을 생성하는 것을 **절대 금지**한다.
문서가 길어질 경우 반드시 파일을 분할하여 순차적으로 작성한다.

---

## STEP 1 — 분량 사전 판단

작성 시작 전, 아래 기준으로 **단일 파일** vs **분할 파일** 여부를 결정한다.

| 조건 | 처리 방식 |
|---|---|
| 예상 줄 수 ≤ 300줄 또는 단일 주제 | 단일 파일로 작성 |
| 예상 줄 수 > 300줄 **또는** 주요 섹션이 3개 이상 | **분할 필수** |
| 복수의 독립적 챕터/모듈/주제 포함 | **분할 필수** |

판단이 애매하면 **분할 방향**으로 결정한다.

---

## STEP 2 — 분할 시 유저 고지 (필수)

분할이 필요하다고 판단되면, **파일을 한 줄도 작성하기 전에** 반드시 다음 형식으로 유저에게 알린다.

```
📄 문서 분량이 많아 여러 파일로 나누어 작성합니다.

분할 계획:
  1. <파일명_1.md> — <주제 요약>
  2. <파일명_2.md> — <주제 요약>
  3. <파일명_3.md> — <주제 요약>
  ...

순서대로 작성하겠습니다.
```

유저의 별도 반대가 없으면 바로 STEP 3으로 진행한다.

---

## STEP 3 — 파일 명명 규칙

### 단일 파일
```
<slug>.md
```

### 분할 파일 (주제 기반 권장)
```
<slug>_<topic>.md         # 예: api_overview.md, api_auth.md, api_errors.md
<slug>_part<N>.md         # 주제 구분이 어려울 때 순번 사용
```

- slug: 영문 소문자 + 하이픈/언더스코어만 사용
- 파일명만 보고 내용을 유추할 수 있어야 한다

---

## STEP 4 — 파일별 작성 원칙

각 파일을 작성할 때:

1. **한 파일씩** 순서대로 `Write` 툴을 호출한다. 병렬 동시 작성 금지.
2. 각 파일은 **자기완결적**으로 작성한다. 독립적으로 읽어도 이해되어야 한다.
3. 파일 상단에 다른 파트와의 관계를 명시한다:
   ```markdown
   <!-- Part 2 of 3 — 이전: api_overview.md / 다음: api_errors.md -->
   ```
4. 한 파일당 **300줄 이하**를 목표로 한다. 어쩔 수 없이 넘어도 **500줄을 절대 초과하지 않는다**.
5. 하나의 `Write` 호출이 끝난 뒤, 다음 파일로 넘어간다.

---

## STEP 5 — 완료 선언

모든 파일 작성 후 반드시 아래 형태로 알린다.

```
✅ 작성 완료

생성된 파일:
  - <파일명_1.md> (<줄 수>줄)
  - <파일명_2.md> (<줄 수>줄)
  - <파일명_3.md> (<줄 수>줄)
```

---

## 금지 행동

- ❌ 300줄 초과 예상 문서를 단일 `Write` 호출 한 번으로 작성
- ❌ 분할 계획 없이 파일부터 쓰기 시작
- ❌ 유저 고지 없이 파일 분할
- ❌ 여러 파일을 병렬로 동시 `Write`
- ❌ 파일 중간에 작업을 멈추고 다음 파일명을 묻기 (계획은 STEP 2에서 한 번에 확정)
